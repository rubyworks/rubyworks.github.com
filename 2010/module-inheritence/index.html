A Case for Module Inheritence

I love beautifully written code. Unfortunately the realities of the language sometimes clash with requirements of the implementation. While Ruby goes a long way toward making beautifully written code common place. It still has some shortcomings I'd like to see improved.

Case in point are my endeaors of the last few days. I've been struggling to re-structure the code of the English project. The primary puporse of the project is to provide extension methods to String, pertaining to the English language. A good exmample are number inflection methods #singluar and #plural. At the same time I am trying to design the library in such a way that other language modules can be built following the same basic structure, and they could all work together if need be.

In the course of this endeavor I decided the best approach, kepping the future squarely in mind, is to create a Language module, upon which English depends, to house all the common logic of (most) langauges. For example, the logic for storing plural and singular special cases might be a good candidate since nearly every language has to account for these. So the relationship between English and Language is cleary one of "inheritence" --I want to encapsulate logic at a general level of abstraction, upon which more specific cases can inherit and augment to fit their particular requirements.

In Ruby this kind of relationship is generally handled by class inheritence. But here I run into an issue. English and Language are modules, and they are modules because they are singleton. Now even though we can include a module into another, which is akin to class inheritence, it does not work quite like class inheritence. For one thing, metaclass methods are not inherited via include. Of course, I could tip-toe around this issue by using extend, since in this case I do not need includable instance methods. But there is a much more subtle issues that raises it's head: the dreaded Double Inclusion Problem. That't a serious headache in my case becuase I want my end users to have the option to load only the components they want to use as needed, rather than forcing them to load the entire system up-front.

So, despite the many calls for the death of the Singleton Pattern, it looks like I will require it for my implmentation of English. Using modules simply comes with too many headaches and hackish work arounds. Now I agree with all the Singleton nay-sayers. It is sort of an anti-pattern to create a class that will only ever have one instantiation and no state, but the trade-off here is worse. If only...

It seems to me the truly fantastic solution to this would be if one module could inherit from another just as we do with classes. And honestly, is there really any reason we shouldn't be able to do this?

  module Language

    def self.words(string)
      # generic definition
    end

  end

  module English < Language

    def self.words(string)
      # override
    end

  end

What is nice about this, something I know that Matz will appreciate, is that it preserves single-inheritence, in contrast to the alternative of turning include into a multiple-inheritence mechinism. And for the love of code beauty, it would mean an end to the all too common:

  def self.included(base)
    base.extend ClassMethods
  end

  module ClassMethods
    # ...
  end

Not only is it ugly, it is even worse for those trying to inherit and extend the behavior. Something more elegant is clearly desireable, and Module Inheritence, as I have demonstrated here, might be just the ticket.
